# investor-watch — 盯盘跟踪技能 设计规格说明书

> 日期：2026-07-16
> 状态：已批准
> 定位：v2.0 核心技能

---

## 一、概述

解决 v1.0 知识库"只写不读"的问题。每次用户问赛道/公司动态时，做增量比对，只输出变化的部分。同时反向更新知识库档案。

### 核心原则

- **不问不跑** — 没有定时任务，用户问了才查
- **增量对比** — 不是每次从头搜，而是和上次快照比差异
- **越用越深** — 每次询问都在更新知识库，下次回答更精准

---

## 二、触发词

| 触发词 | 场景 |
|--------|------|
| "关注[赛道/公司]" | 加入 watch list，记录当前快照 |
| "[赛道/公司]有什么新动态" | 比对更新，输出变化 |
| "最近有什么值得关注的" | 遍历所有已关注条目，汇总变化 |
| "帮我看一下[赛道/公司]" | 查当前快照+历史变化记录 |

---

## 三、工作流

### 模式一：关注

用户说"关注半导体设备赛道"：

```
Step 1: 查知识库
    ├── data/sectors/半导体设备.md → 当前赛道档案
    └── data/companies/ → 该赛道已有公司清单

Step 2: 记录快照
    将当前所有已知数据序列化为快照
    → data/watch/snapshots/半导体设备_20260716.json

Step 3: 写入 checklist
    data/watch/checklist.md 追加一行：
    | 半导体设备 | 赛道 | 2026-07-16 | — | 已关注8家公司 |

Step 4: 告知用户
    "已关注半导体设备赛道（8家公司），有新动态随时问我"
```

### 模式二：查询动态

用户说"半导体设备有什么新动态"：

```
Step 1: 查 watch list
    ├── 确认该赛道在 checklist 中
    ├── 读取上次快照 → snapshots/半导体设备_20260716.json
    └── 读取历史变化 → history.md

Step 2: agent-reach 搜索
    调用 [skill:agent-reach] 搜索最新信息：
    ├── 新融资事件
    ├── 重大新闻/政策
    ├── 上市公司财报/公告
    └── 新入局者

Step 3: 逐条比对
    将搜索结果与上次快照逐项对比：
    ├── 🔴 新信息（快照中没有的）
    ├── 🟡 变化信息（数值/状态有变的）
    └── ✅ 无变化（和快照一致）

Step 4: 输出变化摘要
    📋 [赛道名] 动态追踪（上次检查：2026-07-16）
    
    🔴 新发现（2条）
    ├── 中微公司7月15日发半年报，营收+35%，超预期
    └── SEMI上调2026年半导体设备支出预测至1200亿美元
    
    🟡 估值变化（1条）
    └── 北方华创完成新一轮融资，估值从1200→1500亿
    
    ✅ 其余6家公司无重大变化
    
    💡 建议：中微半年报数据值得深入分析
          → 说"帮我分析一下中微公司"进入深度分析

Step 5: 知识库更新
    ├── 更新 data/companies/中微公司.md（追加半年报数据）
    ├── 更新 data/sectors/半导体设备.md（追加新信息）
    ├── 写入 history.md 变化记录
    └── 写入新快照 snapshots/半导体设备_20260718.json
```

### 模式三：全局一览

用户说"最近有什么值得关注的"：

```
Step 1: 读取 checklist 所有条目
Step 2: 遍历每个条目，按模式二执行
Step 3: 汇总输出

📋 全局动态（共关注 3 个条目）

🔴 半导体设备 — 中微半年报+35%，北方华创融资20亿
✅ 创新药 — 无变化
✅ 固态电池 — 无变化
```

## 四、数据层

```
data/watch/
├── README.md                ← 说明本层用途
├── checklist.md             ← 正在盯什么
│   格式：条目 | 类型(赛道/公司) | 关注时间 | 上次检查 | 最后更新摘要
│
├── history.md               ← 变化记录（追加式）
│   格式：日期 | 条目 | 变化类型 | 摘要 | 影响评级(高/中/低)
│
└── snapshots/               ← 每次检查时的快照（JSON）
    命名规则：[条目名]_YYYYMMDD.json
    内容：关注时的完整数据状态，用于下次增量对比
```

不新增独立目录，watch 本身不持有原始数据，只持有索引和快照。实际数据存在 companies/ 和 sectors/ 中。

---

## 五、存量兼容

已存在的 knowledge base 数据不需要做任何迁移。watch 的 checklist 初始为空，用户说"关注[xxx]"才写入第一次快照。

对于已经分析过的赛道，用户说"关注"时直接取当前知识库内容做快照，无需重新搜索。
