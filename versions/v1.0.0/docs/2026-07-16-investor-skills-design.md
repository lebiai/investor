# 投资人技能套件 — 设计规格说明书

> 日期：2026-07-16
> 状态：已批准
> 作者：Codex Agent

---

## 一、项目概述

为投资人（全品类混合型，零技术背景）打造一套完整的 Codex 实操技能体系，支持开箱即用、纯对话交互。项目完成后可完整迁移至投资人自身的 Codex 环境。

### 核心原则

- **纯对话驱动** — 全部触发词为自然语言，不需要写 prompt 或配置
- **知识库自动累积** — 每看一份材料、每分析一个赛道、每接触一家公司，都写入共用知识库，越用越深
- **模块独立** — 每个 skill 只做一件事，可独立迭代、升级、移除
- **一次迁移，永久使用** — 打包整个项目文件夹即可迁移

## 二、整体架构

```
investor-skills/
├── knowledge-base/                    ← 共享知识库（自动累积，所有 skill 读写）
│   ├── README.md                      ← 知识库说明
│   ├── index.md                       ← 全局内容索引
│   ├── file-index.md                  ← 文件索引（上传时间/文件名/所属项目）
│   ├── companies/                     ← 公司档案
│   ├── sectors/                       ← 赛道档案
│   └── files/                         ← 原始文件归档 + MD 转换件
│       ├── original/                  ← 原始上传文件
│       └── YYYY-MM/                   ← 按月的 MD 转换件
│
├── investor-workbench/                ← 总入口（触发词："开工" / "投资助手"）
├── investor-sector-analysis/          ← 行业研究（触发词："分析一下[赛道]"）
├── investor-research-digest/          ← 信息管理（触发词："帮我整理这个"）
├── investor-content-prod/             ← 内容输出（触发词："帮我写一篇[话题]洞察"）
└── investor-deal-sourcing/            ← 项目 sourcing（触发词："帮我扫描[领域]项目"）
```

## 三、Skill 详细设计

### 3.1 `investor-workbench` — 总入口

**角色**：唯一入口菜单，零基础用户说"开工"就能看到所有能力。

**工作流**：

```
用户 → "开工"
Codex → 打印菜单：
        ① 赛道分析   → "帮我分析一下[赛道]"
        ② 信息整理   → "帮我整理这篇文章/文件"
        ③ 内容产出   → "帮我写一篇洞察"
        ④ 项目扫描   → "帮我扫描[领域]项目"
        ⑤ 知识库浏览 → "打开知识库"（启动 docsify 浏览知识库）
```

**文件结构**：仅一个 SKILL.md（含全部编排逻辑和菜单模板），无子文件夹。

---

### 3.2 `investor-sector-analysis` — 行业研究

**角色**：多机构辩论模式的赛道深度分析，增加公司层扫描。

**工作流**：

```
用户 → "分析一下[赛道名]"

Phase 1: 加载记忆
    ├── 查 knowledge-base/sectors/ → 有无历史分析记录
    └── 如有 → "上次分析过该赛道，新信息已合并"

Phase 2: 六维赛道分析
    ├── 市场规模（TAM/SAM/SOM）
    ├── 增速（历史 CAGR + 未来预测）
    ├── 竞争格局（集中度/头部梯队）
    ├── 技术壁垒（核心技术/护城河）
    ├── 政策环境（监管/补贴/国产替代）
    └── 资本热度（融资事件/估值水位）

Phase 3: 六机构独立评估
    ├── Sequoia / Benchmark / SoftBank / Hillhouse / Buffett / 3G Capital
    ├── 每家输出：核心观点 + 吸引力评分（1-10）
    └── 支持自定义权重（用户说"侧重价值投资"则加大巴菲特权重）

Phase 4: 公司层扫描 ⭐ 新增
    ├── 上市公司地图：按子领域分类，对比营收/增速/毛利率/PE
    ├── 未上市明星项目：融资轮次/估值/投资方
    ├── 产业链位置图：谁卡住哪个环节
    └── 投资信号："龙头优先" / "弹性标的" / "等估值回调"

Phase 5: 辩论引擎
    ├── 共识检测（哪些机构在什么维度上一致）
    ├── 分歧检测（含根因分析 + 严重评级）
    └── 盲点检测（所有机构都忽略的风险因子）

Phase 6: 输出四件套
    ├── 📊 完整分析报告（结构文本）
    ├── 📋 赛道打分卡（各维度 1-10）
    ├── 🔄 辩论对比表（谁赞同/谁反对/为什么）
    └── 📈 公司对比表（上市公司 + 未上市标的）
    
Phase 7: 知识库入库
    ├── 写入 knowledge-base/sectors/
    └── 写入 knowledge-base/companies/（每家新公司单独建档）

结束 → 追问："需要深入分析某家公司吗？"（自然引导 DD 流程）
```

**文件结构**：
```
investor-sector-analysis/
├── SKILL.md                    ← 主编排（含完整工作流）
├── references/
│   ├── analysis-framework.md   ← 六维分析框架定义
│   ├── institutions.md         ← 六家机构人设（内联而非独立文件）
│   ├── company-scan.md         ← 公司层扫描逻辑
│   ├── debate-engine.md        ← 辩论引擎逻辑
│   └── output-specs.md         ← 输出四件规范
└── agents/
    └── openai.yaml
```

### 3.3 `investor-research-digest` — 信息管理

**角色**：全自动文件管理 + 知识沉淀系统。所有上传到 Codex 的文件自动分类、转 MD、入知识库。

**前置依赖**：必须调用微软文件转 MD 能力（Azure Document Intelligence / Office 格式转换 API）。

**工作流**：

```
用户上传文件 / 扔链接 → "帮我整理这个"

Step 1: 文件捕获
    ├── 记录元数据 → knowledge-base/file-index.md
    │   ├── 上传时间
    │   ├── 文件名
    │   ├── 文件类型（PDF/Word/PPT/Excel/网页）
    │   └── 所属项目（AI 根据内容推断）
    └── 原始文件归档 → knowledge-base/files/original/

Step 2: 格式转换
    ├── PDF/Word/PPT/Excel → 微软格式转换 API → MD
    ├── 网页链接 → 爬取 + 转 MD
    └── 原始 MD 持久化 → knowledge-base/files/YYYY-MM/文件名.md

Step 3: 内容分析
    ├── 体裁识别：研报/新闻/财报/纪要/白皮书/其它
    ├── 核心观点提取（3-5 条，标置信度）
    ├── 关键数据提取（数字/增速/市占率/估值）
    └── 关联实体识别（公司/赛道/技术关键词）

Step 4: 知识库关联
    ├── 查 knowledge-base/ → 关联已有公司/赛道
    ├── 更新 knowledge-base/sectors/ 或 companies/ 对应条目
    ├── 如果无匹配 → 新增条目
    └── 更新 knowledge-base/index.md

Step 5: 输出摘要
    ├── 📋 一句话总结（30 字内）
    ├── 📝 核心观点（bullet points）
    ├── 📊 关键数据表
    ├── 🔗 知识库关联链
    └── 💡 补充洞见

Step 6: 文件检索
    用户 → "帮我找一下上个月关于存储芯片的那份报告"
    系统 → 查 file-index.md → 匹配文件名/内容/时间/项目
         → 输出完整 MD 内容
```

**支持批量文档交叉分析**：用户一次投喂 3-5 篇相关文章 → 自动输出跨文档共识/分歧/信息缺口。

**文件结构**：
```
investor-research-digest/
├── SKILL.md                       ← 主编排
├── references/
│   ├── file-processor.md          ← 文件处理规范（格式/元数据/归档）
│   ├── content-analyzer.md        ← 内容分析逻辑（体裁/观点/实体）
│   └── knowledge-base-writer.md   ← 知识库写入协议
└── agents/
    └── openai.yaml
```

### 3.4 `investor-content-prod` — 内容输出

**角色**：基于知识库驱动的多体裁内容生产。不做从零创作，每篇内容根植于分析成果。

**工作流**：

```
用户 → "帮我写一篇关于[话题]的[体裁]"

Step 1: 查知识库
    ├── knowledge-base/sectors/ → 赛道材料
    ├── knowledge-base/companies/ → 公司材料
    └── knowledge-base/file-index.md → 历次研报摘要

Step 2: 用户定调
    ├── 体裁：公众号 / 小红书 / 朋友圈 / 投资备忘录 / 路演纪要
    ├── 角度：行业趋势 / 龙头对比 / 投资机会 / 风险警示
    ├── 长度：短（500字）/ 中（1500字）/ 长（3000字）
    └── 风格：专业严谨 / 通俗易懂 / 犀利观点

Step 3: 内容生成
    ├── 标题（3 个备选，可投喂给社交媒体 AB 测试）
    ├── 正文结构（按体裁模板）
    ├── 数据标注（每项数据对应知识库出处）
    └── 金句抽取（1-2 句可直接分享传播）

Step 4: 保存
    ├── 输出纯文本给用户
    └── 自动写入 outputs/（按 YYYY-MM-DD-标题.md 命名）
```

**预设体裁模板**：

| 体裁 | 字数 | 核心结构 |
|------|------|---------|
| 公众号 | 1500-2500 | 钩子→背景→数据分析→观点→总结 |
| 小红书 | 300-500 | 金句开头→3 个核心点→总结+标签 |
| 朋友圈 | 100-200 | 一句观点+数据支撑 |
| 投资备忘录 | 2000-4000 | 投资逻辑→关键假设→风险→结论 |
| 路演纪要 | 500-1000 | 核心论点→3-5 条论据→下一步 |

**文件结构**：
```
investor-content-prod/
├── SKILL.md
├── references/
│   ├── template-library.md       ← 各体裁模板定义
│   └── content-quality-gate.md   ← 内容质量检查清单
└── agents/
    └── openai.yaml
```

### 3.5 `investor-deal-sourcing` — 项目扫描

**角色**：在赛道基础上，实时扫描新项目/新融资/新入局者，持续更新项目池。

**工作流**：

```
用户 → "帮我扫描一下[领域]有什么新项目"

Step 1: 查知识库
    ├── knowledge-base/sectors/[领域].md
    ├── knowledge-base/companies/ → 已关注标的最新动态
    └── 标记已知 vs 未知

Step 2: 实时扫描
    ├── 最近 3-6 个月新融资事件
    ├── 新入局者/跨界玩家
    ├── 标的估值区间
    └── 产业资本 + 财务资本布局情况

Step 3: 输出项目清单
    ├── 📋 已关注意向项目（知识库已有 + 最新动态更新）
    ├── 🔍 新发现项目（最新融资/公开报道）
    ├── 📊 估值参考（同赛道可比区间）
    ├── 🔗 人脉线索（如果搜到创始团队/投资机构信息）
    └── 💡 跟进建议（优先看哪个、为什么）

Step 4: 知识库入库
    ├── 写入 knowledge-base/companies/（新公司建档）
    ├── 更新 knowledge-base/sectors/ 对应条目
    └── 标注发现日期 + 来源
```

**文件结构**：
```
investor-deal-sourcing/
├── SKILL.md
├── references/
│   ├── scan-protocol.md          ← 扫描流程规范
│   └── project-template.md       ← 项目档案模板
└── agents/
    └── openai.yaml
```

## 四、知识库设计 (`knowledge-base/`)

### 4.1 设计原则

- **所有 skill 共享** — 不分家，统一写、统一查
- **SKILL.md 驱动写入** — 每个 skill 在各自编排逻辑中定义"什么情况写入什么"
- **不做独立的"知识库维护 Skill"** — 知识库是副产品，不是独立功能
- **目录即导航** — index.md 是全局索引，file-index.md 是文件索引

### 4.2 目录结构

```
knowledge-base/
├── README.md                      ← 说明文档：这是自动累积的知识库
│                                   结构说明 + 使用方式
│
├── index.md                       ← 全局索引（所有 skill 自动更新）
│   ├── 赛道索引：[赛道名] → 分析时间 / 访问路径
│   ├── 公司索引：[公司名] → 建档时间 / 来源分析 / 访问路径
│   └── 标签索引：[关键词] → 关联条目
│
├── file-index.md                  ← 文件索引（仅 research-digest 写入）
│   记录格式：
│   | 上传时间 | 文件名 | 文件类型 | 所属项目 | MD路径 |
│   |----------|--------|---------|---------|--------|
│   | 2026-07-16 | 大摩半导体研报.pdf | 研报 | 半导体设备 | files/2026-07/xxx.md |
│
├── companies/                     ← 公司档案
│   ├── README.md                  ← 说明
│   ├── 中微公司.md                ← 格式见下方 4.3
│   ├── 北方华创.md
│   └── ...
│
├── sectors/                       ← 赛道档案
│   ├── README.md                  ← 说明
│   ├── 半导体设备.md              ← 格式见下方 4.4
│   └── ...
│
└── files/                         ← 文件存档（仅 research-digest 写入）
    ├── original/                  ← 原始上传文件
    ├── 2026-07/                   ← 按月归档的 MD 转换件
    └── ...
```

## 五、知识库浏览

使用 **docsify** 运行极简知识库站点：

```
cd knowledge-base
npx docsify init ./    ← 生成 index.html
npx docsify serve .     ← 本地启动，浏览器打开即可浏览
```

所有 MD 文件自动渲染为可搜索的知识库网站，零配置。

在 `investor-workbench` 中展示入口提示：

```
⑤ 知识库浏览 → 终端运行以下命令：
   cd investor-skills/knowledge-base && npx docsify serve .
   然后在浏览器打开 http://localhost:3000
```

## 六、数据流全览

```
                    ┌──────────────────────┐
                    │      用户对话输入       │
                    └──────┬───────────────┘
                           │
                    ┌──────▼───────────────┐
                    │   investor-workbench  │
                    │   (总入口 / 菜单)       │
                    └──────┬───────────────┘
            ┌──────────────┼───────────────┐
            │              │               │
    ┌───────▼──────┐ ┌────▼─────┐  ┌──────▼──────┐
    │ sector-      │ │research- │  │deal-        │
    │ analysis     │ │digest    │  │sourcing     │
    └───────┬──────┘ └────┬─────┘  └──────┬──────┘
            │             │               │
            └─────────────┼───────────────┘
                          │
                    ┌─────▼──────┐
                    │ knowledge- │ ←── content-prod 也读这里
                    │    base    │
                    └────────────┘
```

- **写入口**：sector-analysis、research-digest、deal-sourcing 会写入 knowledge-base
- **读入口**：content-prod、deal-sourcing、sector-analysis（历史复用）会读取 knowledge-base
- **无写冲突**：每个 skill 写入不同区域（companies/ vs sectors/ vs files/），互不覆盖

## 七、部署方案

完成所有 skill 搭建后，执行：

```bash
# 1. 删除仅与本环境相关的文件（如 .env 中的 API key）
# 2. 打包
tar czf 投资人.tar.gz 投资人/

# 3. 在投资人 Codex 环境解压
# 4. 投资人使用触发词即可使用，无需额外配置
```

每个 skill 的 openai.yaml 中声明触发词，Codex 自动识别。

## 八、非功能性要求

- **纯对话交互**：投资人不说 trigger word 就不激活对应 skill，不影响日常对话
- **容错**：如果某个 skill 调用中知识库不可用，降级为纯生成模式，不中断用户体验
- **渐进式迁移**：可以先迁移 workbench + sector-analysis，逐步加入其它 skill
