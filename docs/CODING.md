# 操作规范（Agent 专用）

> 本文件定义 Agent 在此项目上的执行纪律。目的是避免碎片化操作、中间态残留和不必要的来回迭代。
> 用户不需要了解这些细节，Agent 必须遵守。

## 1. 操作原则

### 1.1 目标态驱动

修改前先想清楚最终结果长什么样，用 `cat > file` 或 `python3 << PYEOF` 一次到位，禁止分成多步迭代。

```
❌ 先改 path → 再改版本 → 再修前缀 → 再修显示
   4 步操作，每步一个单独命令
✅ 一次 cat > file，包含所有变更
```

### 1.2 批次执行，不逐个文件

涉及多个同类改动（如给 8 个 SKILL.md 加 version 字段），用循环/脚本一次搞定，禁止一个个文件手动操作。

```
✅ for skill in skills/*/; do python3 <<< "${skill}..."; done
❌ 先改 sector-analysis, 再改 research-digest, 再...
```

### 1.3 读文件并发

多个独立文件需要读时，用单次 `cat file1 && echo === && cat file2` 或 `python3 + glob`，禁止一个文件一个 cat 地等。

### 1.4 中间态清零

如果第一次操作失败/中断，在开始第二次之前清除第一次留下的残留，避免两份叠加。

```
❌ head -n -9 失败 → 直接 python append → 数据重复
✅ head -n -9 失败 → 确认 CHANGELOG 当前状态 → 清除 → 重写
```

## 2. Shell 纪律

### 2.1 文件写入用 `cat > file`，不用 `sed -i`

| 场景 | 用 | 不用 |
|------|----|------|
| 创建新文件 | `cat > file << 'EOF'` | `echo ... > file` |
| 重写整个文件 | `cat > file << 'EOF'` | 多段 sed |
| 批量改路径前缀 | `python3` | `sed -i` |
| 插一行到文件中间 | `python3 读行→插入→写回` | `sed -i 'x,ys/.../.../'` — 跨平台兼容差 |

**为什么：** `sed -i` 在 macOS/Linux 语法不同，且大段替换时不可逆。Python 可控、可读、跨平台。

**例外：** 单次明确模式替换（如 `sed -i '' 's/旧函数/新函数/g'`），且用 `rg` 验证后再提交。

### 2.2 参数扩展限制

以下通配符和变量扩展在 shell 中有跨行/转义风险：

| 场景 | 问题 | 解决方案 |
|------|------|---------|
| heredoc 内嵌反引号 | 解释为命令替换 | 用 `'HEREDOC_MARKER'` 阻止扩展 |
| heredoc 内嵌 $ | 解释为变量替换 | 同上 |
| 复杂字符串的 sed | 特殊字符大量转义 | 改用 python3 |

### 2.3 macOS 兼容

| 命令 | macOS 行为 | 替代 |
|------|-----------|------|
| `head -n -5` | **不支持** 负数 | `python3 -c "lines=open(...); open('w').writelines(lines[:-5])"` |
| `sed -i ''` | 必须加空串 | 避免使用，用 python3 替代 |
| `date` 格式 | `-d` 不存在 | `date -j -f` 或 `python3 -c "from datetime import..."` |

## 3. 验证纪律

### 3.1 先定标准，再验证

不是改完再想怎么查。而是在改之前就明确验证标准：

```
✅ 改之前：
  预期结果：8 个 SKILL.md 都有 version: x.y.z
  验证命令：rg '^version: ' skills/*/SKILL.md | wc -l = 8

❌ 改之后：
  查一下...看看有没有问题...再看看那里...
```

### 3.2 单次验证，不反复查

在一轮操作结束时，运行一次完整的验证（路径检查 / 版本号 / 文件完整性），而不是每改一个文件就验证一次。

但如果某步是**不可逆操作**（如 `rm -rf`、`mv` 覆盖已有文件），执行后立即验证该步。

## 4. 本地化检查清单

每次涉及以下操作时，对照执行：

- [ ] 目标态想清楚了？还是一个文件一个文件试？
- [ ] 多个同类改动能用循环/批量一次搞定？
- [ ] 文件重写 > sed 逐行改？
- [ ] 如果用 Python heredoc，确认不踩反引号和 $ 的坑？
- [ ] macOS 兼容性考虑了吗？（`head -n -5`、`sed -i`）
- [ ] 改前有预想的验证标准？还是改完再想怎么查？

## 5. 事故回放参考

以下是从本项目的实际教训中提取（用于理解为什么要有这些规则，不保留在最终描述中）：

| 事故 | 违反规则 | 教训 |
|------|---------|------|
| 路径修了 3 轮才干净 | 1.1 目标态驱动 | 改之前就应该确认所 ../../data/ 和 ../../../data/ 的分布 |
| CHANGELOG 重复 | 1.4 中间态清零 | head 失败后直接追加，没先清理 |
| archive 目录名从 latest 改到 v1.0.0 到 v1.0.0 不一致 | 1.1 目标态驱动 | 迁 archive 前没想清楚最终目录结构 |
| sed 在 workbench 菜单失败 | 2.1 文件写入 | 特殊字符的 sed 不可靠，应该 cat > file 覆盖 |
| rg 中文匹配失败 | — | rg 和 grep 对中文/特殊字符行为不同，优先用 python3 处理 |
